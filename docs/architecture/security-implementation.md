## Purpose

Capture the security and privacy posture that actually exists today, plus the controls that are still missing before any production rollout.

## Current Security Thesis

The client is not trusted.

That principle is already visible in the current codebase:

- ownership is checked server-side
- document job and artifact reads are owner-scoped
- policy blocks are generated by the backend
- generic event metadata is validated and redacted
- retention rules are enforced in backend flows, not guessed in the client

## Implemented Controls

### Ownership and resource checks

Current runtime enforces ownership around:

- user sessions
- jobs
- document analyses
- document artifacts
- feedback endpoints

### Document protections

Implemented today:

- upload content-type validation
- payload size checks
- per-session artifact quotas
- per-session active-job quotas
- `PolicyRejection` as the stable block contract
- idempotency fingerprint checks
- artifact deletion and artifact expiration after terminal job states

### Event safety

Event metadata validation already protects against generic leakage of:

- raw document text
- full OCR text in event metadata
- secrets and tokens
- other forbidden metadata keys

### PWA protection

The current service worker is hardened for:

- read-only shell caching
- no mutation replay
- no caching of sensitive document payloads

## Important Current Limitation

Authentication is still local-development grade.

The current app uses local user scoping and request headers for ownership checks, which is acceptable for localhost simulation and tests, but not for real users.

## Data Retention Posture

Current behavior:

- uploaded blobs can exist briefly for processing
- terminal document jobs expire the raw artifact blob
- history keeps derived analysis, not the full raw document body
- event metadata keeps references and counters, not raw content

This is a meaningful step forward, but it is still not a full production privacy program.

## What Progressed

Recently completed:

- stable `PolicyRejection` contract
- upload and job quota visibility
- artifact expiration after worker and manual terminal transitions
- `blocked_reason` read model support
- safer offline shell boundaries

## What Still Must Be Built

Before any production release:

- real authentication
- real session management
- production authorization test matrix
- secret management beyond local env handling
- stronger audit coverage for consent/privacy settings
- browser security review of PWA flows
- encryption/storage review for persisted local artifacts on non-dev machines

## Current Risk Summary

Strong enough for localhost MVP:

- backend ownership checks
- retention-aware document flow
- event redaction
- explicit policy blocks

Not strong enough for production:

- identity model
- auth tokens/session lifecycle
- hardened deployment posture
- incident/audit tooling

## Recommended Next Security Work

1. Replace local header auth with a real auth boundary.
2. Add authorization-focused HTTP tests for cross-user access.
3. Define explicit consent flows if more sensitive modules are added.
4. Add browser E2E checks for document deletion, quota blocks, and offline behavior.
